# Runbook: Regime Classifier

The regime classifier assigns one of four market labels based on SPY-derived
features. Downstream signal generators and decision gates use this label to
filter trades: for example, momentum strategies may be disabled in `bear-trend`
or `high-vol` regimes.

## How it works

### Current path: rule-based (default)

Four labels, three features, no model required:

| Label | Rule |
|-------|------|
| `high-vol` | `spy_20d_vol > 0.30` (annualised) |
| `bull-trend` | `spy_20d_return > 0.03` AND `sma_50 > sma_200` |
| `bear-trend` | `spy_20d_return < -0.03` AND `sma_50 < sma_200` |
| `choppy` | Everything else |

Rules are evaluated in the order shown. `high-vol` wins over any trend condition.

Feature definitions:

- `spy_20d_return`: price return over the last 21 close prices (i.e. 20 trading days)
- `spy_20d_vol`: annualised daily log-return vol over the last 20 trading days, ddof=1
- `sma_50_above_200`: True when the 50-day simple moving average is above the 200-day SMA

Minimum observations: 210 bars are required before any feature can be fully computed.
Below that threshold the classifier returns a conservative default.

### Optional path: XGBoost (feature-flagged)

Set `TRADER_REGIME_USE_XGB=true` in the engine `.env` to activate. The engine loads
`models/regime_xgb.joblib` on the first classify call (lazy, thread-safe). The
XGBoost model uses the same four labels but with six features:

```
spy_20d_return, spy_20d_vol, sma_50_above_200,
spy_60d_return, spy_60d_vol, return_z_20d
```

Feature order is read from `models/regime_xgb.meta.json` at load time. If the
model loads but inference fails on any single call, the classifier silently falls
back to the rule-based path for that request.

The rule-based path is always the fallback: flag off, model file missing, or any
load / inference error all land here.

## Swapping in a new XGBoost model

### 1. Train

From the engine repo root with Alpaca credentials in `.env`:

```bash
python scripts/train_regime.py
# Options:
python scripts/train_regime.py --symbol SPY --days 1825 --out models/regime_xgb.joblib
```

The script performs a chronological 70/15/15 walk-forward split. Check the output
for accuracy on the val and test splits before promoting. Ballpark expectation for a
rule-bootstrapped model: >85% agreement with the rule-based classifier on the test
split.

The script writes two files:

- `models/regime_xgb.joblib` -- the model wrapped with `feature_list` and `label_map`
- `models/regime_xgb.meta.json` -- the sidecar that `regime.py` reads at inference

Both files must be present. The sidecar is the authoritative source for feature order
and label mapping.

### 2. Validate

Confirm that the meta JSON reflects the expected feature list:

```bash
cat models/regime_xgb.meta.json | python3 -m json.tool
```

Expected keys: `feature_list` (array of 6 feature names in training order),
`label_map` (dict mapping integer class index to label string).

Run a sanity-check inference before toggling the flag:

```bash
TRADER_REGIME_USE_XGB=true python3 -c "
from trader_engine.signals.regime import classify_regime, compute_regime_features
import pandas as pd, numpy as np
spy = pd.Series(np.linspace(400, 420, 250))  # synthetic rising SPY
features = compute_regime_features(spy)
print(features)
print('regime:', classify_regime(features))
"
```

### 3. Deploy

```bash
# Set the flag in the engine .env:
echo "TRADER_REGIME_USE_XGB=true" >> .env

# Restart the engine:
sudo systemctl restart trader-engine
# or on macOS:
launchctl bootout gui/$(id -u) ~/Library/LaunchAgents/com.trader-engine.plist
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.trader-engine.plist
```

The first classify call after restart will load and cache the model. Check logs for
`regime.path mode=xgb` to confirm the XGB path is active.

### 4. Rollback

```bash
# Remove or comment the flag:
# In .env: TRADER_REGIME_USE_XGB=false (or delete the line)
# Then restart the engine.
```

No model artifact changes needed. The rule-based path has no external dependencies.

You can also use the env var override without editing `.env`:

```bash
TRADER_REGIME_USE_XGB=false uvicorn trader_engine.api.main:app ...
```

## Rule-based threshold tuning

All rule thresholds live in `_classify_rule_based()` in `regime.py`:

```python
if vol > 0.30:          # high-vol threshold
    return 'high-vol'
elif ret > 0.03 and trend:    # 3% 20d return + golden cross
    return 'bull-trend'
elif ret < -0.03 and not trend:
    return 'bear-trend'
```

Adjustments should be tested against at least 2 years of historical SPY data before
deploying. Note that the training labels for the XGBoost model are generated by this
same function, so changing these thresholds means you need to retrain the XGBoost
model to stay aligned with the rules.

## Verification

**Check the active path from engine logs:**

```bash
# After restart, first classify call logs the path:
journalctl -u trader-engine | grep "regime.path"
# Expect: regime.path mode=rule-based  (or xgb if flag is on)
```

**Spot-check current regime via the scheduler output:**

The scheduler logs the current regime before each signal generation cycle. Look for
`regime_label` in structured log output:

```bash
journalctl -u trader-engine --since "today" | grep "regime_label"
```

**Check for XGBoost load failures:**

```bash
journalctl -u trader-engine | grep -E "regime.xgb\.(model_missing|load_failed|joblib_import_failed)"
```

Any of these means the engine fell back to rule-based (expected and safe).

**Model sidecar drift check** (after any training run):

```bash
python3 -c "
import json
with open('models/regime_xgb.meta.json') as f:
    meta = json.load(f)
print('Features:', meta['feature_list'])
print('Labels:', meta['label_map'])
assert len(meta['feature_list']) == 6, 'expected 6 features'
assert set(meta['label_map'].values()) == {'bull-trend','bear-trend','choppy','high-vol'}
print('OK')
"
```

## Shadow evaluation and flip procedure

Phase 5 Task 6 wired shadow logging into `classify_regime`. From this build
forward, every classify call computes BOTH the rule-based label and the
XGBoost label (when the model is loadable) and appends a row to a SQLite
log regardless of which label is returned to the caller. The
`TRADER_REGIME_USE_XGB` flag still controls which label flows through to
the trader; the log accumulates parallel-evaluation evidence so the flip
happens on data, not on a hunch.

### Where the log lives

Default path: `models/regime_predictions.db` under the engine working
directory. Override with `TRADER_REGIME_PREDICTIONS_LOG_PATH=/abs/path` or
the `regime_predictions_log_path` settings field.

Schema:

```
regime_predictions (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  ts_ms         INTEGER NOT NULL,
  rule_label    TEXT NOT NULL,
  xgb_label     TEXT,        -- nullable; NULL when XGBoost shadow failed
  features_json TEXT NOT NULL
)
INDEX idx_regime_pred_ts ON regime_predictions(ts_ms DESC)
```

`xgb_label` is nullable on purpose: when the model file is missing, when
joblib import fails, or when `predict()` raises, the row still records
the rule-based label that the engine actually returned and the XGBoost
column is left NULL. Filter on `xgb_label IS NULL` to spot shadow
outages.

### Fast path: scripts/regime-shadow-report.sh

Phase 6 Task 1 shipped a one-shot diagnostic that runs all three
queries below against the engine DB over SSH and prints a PASS / FAIL
verdict tied to the flip criteria. Exit code is 0 on PASS, 1 on FAIL
or inconclusive (fewer than 100 samples).

```bash
# 4-week flip window (default):
scripts/regime-shadow-report.sh

# Weekly health check:
scripts/regime-shadow-report.sh 7
```

Env overrides if the engine host or DB path change: `ENGINE_HOST`,
`ENGINE_USER`, `REGIME_DB_PATH`.

### Manual: compute the agreement matrix

```bash
sqlite3 ~/Projects/trader-engine/models/regime_predictions.db <<'SQL'
SELECT
  rule_label,
  xgb_label,
  COUNT(*) AS n
FROM regime_predictions
WHERE ts_ms > strftime('%s','now','-7 days') * 1000
GROUP BY 1, 2
ORDER BY n DESC;
SQL
```

The `* 1000` converts SQLite's seconds-epoch (`strftime('%s')`) into the
millisecond timestamps stored in `ts_ms`. The diagonal (rule_label =
xgb_label) is your agreement count; off-diagonal cells are disagreements
broken out by direction (e.g. rule says bull-trend, xgb says choppy).

Quick agreement-rate one-liner:

```bash
sqlite3 ~/Projects/trader-engine/models/regime_predictions.db <<'SQL'
SELECT
  ROUND(100.0 * SUM(CASE WHEN rule_label = xgb_label THEN 1 ELSE 0 END)
        / COUNT(*), 2) AS agreement_pct,
  COUNT(*) AS samples
FROM regime_predictions
WHERE ts_ms > strftime('%s','now','-7 days') * 1000
  AND xgb_label IS NOT NULL;
SQL
```

### Flip procedure (after 4 weeks of parallel evaluation)

Run only if BOTH conditions hold:

1. Majority-class agreement is above 95% over the trailing 4-week window.
2. No silent divergence on regime transitions. Spot-check:
   ```bash
   sqlite3 .../regime_predictions.db <<'SQL'
   SELECT ts_ms, rule_label, xgb_label
   FROM regime_predictions
   WHERE rule_label != xgb_label
     AND ts_ms > strftime('%s','now','-7 days') * 1000
   ORDER BY ts_ms DESC LIMIT 50;
   SQL
   ```
   No clusters of one-sided disagreement (e.g. xgb stuck on `high-vol`
   while rule says `bull-trend` for two solid days). Anything that looks
   systemic delays the flip and triggers a retrain.

When ready:

```bash
# 1. Set the flag in the engine env file:
sudo $EDITOR /etc/trader-engine.env
# Add or change: TRADER_REGIME_USE_XGB=true

# 2. Restart the service:
sudo systemctl restart trader-engine

# 3. Verify live behaviour. The first /signals/regime call after the
#    restart should be served by the XGBoost path:
curl -H "X-Engine-Token: $TOKEN" http://your-engine-host:8200/signals/regime

# 4. Confirm the path in logs:
journalctl -u trader-engine | grep "regime.path" | tail -1
# Expect: regime.path mode=xgb
```

### Rollback

If anything looks wrong after the flip, the rollback is one env edit and
one restart:

```bash
sudo $EDITOR /etc/trader-engine.env
# Either remove the line or set TRADER_REGIME_USE_XGB=false

sudo systemctl restart trader-engine
```

The model artifact stays loaded but is bypassed -- shadow logging still
writes both labels, so the next evaluation window starts collecting data
immediately. No artifact changes required.

### Verification: log is recording rows

```bash
# Should grow monotonically as the scheduler ticks fire classify_regime:
sqlite3 ~/Projects/trader-engine/models/regime_predictions.db \
  "SELECT COUNT(*) FROM regime_predictions"

# Most recent 5 rows for a sanity check:
sqlite3 ~/Projects/trader-engine/models/regime_predictions.db \
  "SELECT ts_ms, rule_label, xgb_label FROM regime_predictions
   ORDER BY ts_ms DESC LIMIT 5"
```

If the row count is not increasing, check engine logs for
`regime.shadow_log.append_failed` -- the log layer is wrapped in a try /
except so a corrupt file or permissions issue cannot stall inference,
but it will silently stop recording until cleared.
